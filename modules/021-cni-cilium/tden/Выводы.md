Развертывание Deckhouse в Yandex Cloud (модуль `cloud-provider-yandex`) позволяет использовать нативные возможности облачной сети VPC. Благодаря тому, что Cloud Controller Manager (CCM) автоматически управляет таблицами маршрутизации в облаке, обеспечивается полная L3-связность между всеми узлами кластера. Это позволяет настроить Cilium максимально эффективно, исключив лишние уровни инкапсуляции.

Ниже представлены рекомендованные настройки модуля **cni-cilium**, оптимизированные для работы в инфраструктуре Yandex Cloud:

### 1. Настройка маршрутизации и инкапсуляции

- `tunnelMode: "Disabled"`
  - **Почему:** В Deckhouse компонент `cloud-controller-manager` (CCM) запускается с флагом `--configure-cloud-routes=true` и автоматически синхронизирует выделенные узлам подсети (`node.spec.podCIDR`) с облачной таблицей маршрутизации (VPC Route Table). В облаке создаются статические записи, где целевым префиксом (Destination) является подсеть подов конкретного узла, а следующим Next Hop — внутренний IP-адрес этого узла.
  - **Результат:** Пакеты между подами передаются как обычный IP-трафик через стандартный шлюз узла. Это исключает накладные расходы на инкапсуляцию (VXLAN), сохраняет нативный MTU (1500 байт) и снижает нагрузку на CPU, так как узлам не нужно выполнять инкапсуляцию и деинкапсуляцию трафика.

- `createNodeRoutes: false`
  - **Почему:** В облаке Yandex Cloud за связность между подсетями (зонами) отвечает Route Table в VPC. Если включить этот параметр, Cilium будет пытаться создать прямые маршруты через соседние узлы. Это эффективно только в рамках одного L2-сегмента, но в многозонном кластере узлы в разных зонах доступности не имеют прямой видимости.
  - **Результат:** Гарантируется стабильная связность между всеми узлами кластера независимо от их размещения. Трафик между узлами (даже в одной подсети) будет проходить через виртуальный роутер облака (Default Gateway). В современных SDN-сетях такой "прыжок" через шлюз не создает значимых задержек, но зато полностью исключает проблемы с недоступностью подов в разных зонах доступности.

### 2. Балансировка и защита

- `bpfLBMode: "Hybrid"`
  - **Почему:** Отсутствие инкапсуляции (Direct Routing) позволяет Cilium использовать технологию DSR (Direct Server Return) для TCP-трафика. В режиме `Hybrid` Cilium применяет DSR для TCP и SNAT для UDP. Когда узел кластера принимает входящий запрос на Service IP, он пересылает его целевому поду без подмены адреса отправителя. Ответный трафик от пода проходит через сетевой стек своего узла и уходит напрямую инициатору через роутер облачной сети YC.
  - **Результат:** При использовании сетевого балансировщика (L4 NLB) полностью сохраняется исходный IP-адрес отправителя (`Source IP`) — будь то пользователь из Интернета или информационная система в другой сети. Это исключает лишний «прыжок» через узел-балансировщик на обратном пути, снижая задержки (`latency`) и нагрузку на сеть.

- `svcSourceRangeCheck: false`
  - **Почему:** В типовых сценариях использования Yandex Cloud доступ к кластеру организуется через облачные балансировщики (ALB/NLB). В этих условиях фильтрацию входящего трафика по IP-адресам целесообразно делегировать инфраструктурному уровню — облачным Security Groups. Это позволяет отсекать нежелательные пакеты на периметре VPC. Кроме того, при использовании Application Load Balancer (ALB) для публикации HTTP-приложений облако всегда выполняет SNAT. В такой схеме локальная проверка Cilium на узле увидит IP-адрес балансировщика вместо адреса клиента и заблокирует легитимный трафик. Отключение параметра является «безопасным дефолтом» для облачной среды.
  - **Результат:** Гарантируется стабильная работа сервисов в любой схеме доступа (включая ALB со SNAT) и экономится ресурс CPU на узлах за счет удаления избыточных eBPF-проверок.

### 3. Masquerade и внешние IP-адреса

- `masqueradeMode: "BPF"`
  - **Почему:** Режим eBPF-маскарадинга значительно производительнее классического Netfilter (iptables). Специфика Yandex Cloud заключается в том, что публичные ExternalIP не назначаются на сетевые интерфейсы узлов, а работают через NAT 1:1 на стороне облака. eBPF-программы Cilium «видят» только внутренние адреса узлов, но этого достаточно для корректного маскарадинга трафика, выходящего в интернет.
  - **Результат:** Обеспечивается высокая пропускная способность и минимальные задержки. Тот факт, что Cilium не знает о существовании ExternalIP, не создает проблем, так как всё внутреннее взаимодействие в Deckhouse (между подами и узлами) по умолчанию происходит через внутреннюю сеть YC. Режим `"BPF"` является наиболее современным и рекомендуемым.

### 4. Оптимизация (Для крупных инсталляций)

- `labelsRegex`
  - **Почему:** В больших кластерах множество уникальных комбинаций лейблов (например, хеши в именах подов) вызывает избыточную генерацию идентификаторов Cilium (Security Identities), что повышает нагрузку на память и CPU контроллера.
  - **Результат:** Данный параметр **не включается в базовую рекомендованную конфигурацию**, так как оптимальный набор исключаемых лейблов индивидуален для каждого проекта. Пользователям рекомендуется настраивать его самостоятельно при значительном росте количества объектов в кластере.

### Итоговый рекомендуемый манифест (`cni-cilium` ModuleConfig)

Данная конфигурация является оптимальным выбором для старта Production-ready кластера в Yandex Cloud:

```yaml
apiVersion: deckhouse.io/v1alpha1
kind: ModuleConfig
metadata:
  name: cni-cilium
spec:
  enabled: true
  settings:
    # Использование нативной VPC-маршрутизации (без VXLAN)
    tunnelMode: "Disabled"
    # Высокая производительность при выходе во внешние сети
    masqueradeMode: "BPF"

    # --- Ниже перечислены параметры, которые являются дефолтными в Deckhouse. ---
    # --- Их можно не указывать, так как на этапе начальной загрузки кластера (bootstrap) ---
    # --- механизм обратной совместимости (d8-cni-configuration) их не обрабатывает: ---

    # Связность между зонами обеспечивает Route Table в YC
    # createNodeRoutes: false

    # Оптимизация для L4 NLB (сохранение SourceIP и снижение latency)
    # bpfLBMode: "Hybrid"

    # Фильтрация делегирована облачным Security Groups
    # svcSourceRangeCheck: false
```

> **Примечание:** Параметр `labelsRegex` рекомендуется настраивать индивидуально для крупных кластеров с целью оптимизации ресурсов контроллера.
