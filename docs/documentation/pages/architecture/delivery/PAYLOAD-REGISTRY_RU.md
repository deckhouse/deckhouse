Модуль [`payload-registry`](/modules/payload-registry/) реализует пользовательское хранилище образов контейнеров, соответствующее стандартам OCI, внутри Deckhouse Kubernetes Platform (DKP).

Подробнее с настройками модуля и примерами его использования можно ознакомиться в [соответствующем разделе документации](/modules/payload-registry/).

## Архитектура модуля

{% alert level="info" %}
Для упрощения схемы приняты следующие допущения:

* На схеме показано, что контейнеры разных подов взаимодействуют друг с другом напрямую. Фактически они взаимодействуют через соответствующие сервисы Kubernetes (внутренние балансировщики). Названия сервисов не указываются, если они очевидны из контекста. В остальных случаях название сервиса указано над стрелкой.
* Поды могут быть запущены в нескольких репликах, однако на схеме все поды изображены в одной реплике.
{% endalert %}

Архитектура модуля [`payload-registry`](/modules/payload-registry/) на уровне 2 модели C4 и его взаимодействие с другими компонентами DKP изображены на следующей диаграмме:

## Компоненты модуля

Модуль состоит из следующих компонентов:

1. **Apiserver** (Deployment) — компонент состоит из одного контейнера **apiserver** и реализует расширение API Kubernetes для получения информации о содержимом хранилища образов контейнеров. Для этого можно использовать [`Deckhouse CLI (d8)`](/products/kubernetes-platform/documentation/v1/cli/d8/) или `kubectl`.

   Все операции выполняются с использованием виртуального типа объектов Kubernetes `payloadrepositorytags.payload-registry.deckhouse.io` (PayloadRepositoryTag). Эти объекты нельзя создать или изменить с использованием API Kubernetes. Они отображают содержимое хранилища образов контейнеров.

   При запросе apiserver обращается к компоненту registry и выполняет одну из следующих операций:

    * ListRepositories — читает каталог репозиториев;
    * ListTags — читает список тегов в репозитории;
    * GetTagInfo — читает манифест, конфиг и дайджест (digest) конкретного тега;
    * DeleteTag — удаляет тег.

   {% alert level="info" %}
   При удалении тега apiserver выполняет следующие действия:
    * проверка на существование тега;
    * заливка технического образа в этот тег;
    * удаление получившегося манифеста по дайджесту.

   Это обеспечивает удаление только указанного тега, а не всех тегов, ссылающихся на этот же образ.
   {% endalert %}

1. **Pullsecrets-controller** (Deployment) — контроллер, состоящий из одного контейнера **controller**, обеспечивает обновление ресурса Secret с токеном доступа к registry.

   Pullsecrets-controller выполняет следующие действия:

    * следит за Secret типа `kubernetes.io/dockerconfigjson`, у которых присутствует аннотация `payload-registry.deckhouse.io/service-account.name`;
    * запрашивает токен для указанного ServiceAccount;
    * формирует `.dockerconfigjson` и записывает его в Secret;
    * периодически проверяет срок жизни токена и обновляет его заранее;
    * при удалении Secret убирает ссылку на него из ресурса ServiceAccount;
    * при удалении ServiceAccount удаляет связанные с ним ресурсы Secret.

1. **Registry** (Deployment) — основной компонент, реализующий управление хранилищем образов на основе [Docker Distribution](https://github.com/deckhouse/3p-distribution).

   В качестве бэкенда хранения registry поддерживает:
    * файловую систему;
    * S3-совместимое хранилище;
    * оперативную память.

   Кроме того registry может проксировать запросы к другим хранилищам образов с возможностью кеширования данных.

   Состоит из следующих контейнеров:

    * **registry** — основной контейнер;
    * **auth** — сайдкар-контейнер с сервисом аутентификации и авторизации доступа к реестру образов.

1. **Registry-gc-&lt;uid&gt;** (Job) — компонент, состоящий из одного контейнера **gc**, выполняет удаление нетегированных образов.

   Задача запускается по расписанию хуком модуля, который выполняет следующие действия в указанном порядке:

    * проверяет настройки и расписание, определяет требуется ли сейчас запускать компонент. Расписание задаётся в параметре модуля [`.settings.gc.schedule`](/modules/payload-registry/stable/configuration.html#parameters-gc-schedule);

    * переводит registry в режим только для чтения. Хук перезапускает Deployment `registry` с параметром `internal.gc.run.readOnly=True` и дожидается готовности пода;

    * создаёт Job с командой `registry garbage-collect /config/distribution_config.yaml --delete-untagged` и дожидается завершения работы задачи;

    * переводит registry в режим read-write (RW), аналогично изменяя шаблон Deployment.

## Взаимодействия модуля

Модуль взаимодействует со следующими компонентами:

* **Kube-apiserver**:
  * аутентификация и авторизация доступа к хранилищу образов;
  * отслеживание и обновление ресурсов Secret и ServiceAccount.

С модулем взаимодействуют следующие внешние компоненты:

1. **Kube-apiserver** — выполнение запросов к ресурсам API-группы `payload-registry.deckhouse.io`.

1. **Ingress-контроллер** — пересылка запросов пользователя на скачивание или загрузку образов.
