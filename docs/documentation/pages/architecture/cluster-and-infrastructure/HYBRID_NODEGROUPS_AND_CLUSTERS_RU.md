Следует понимать разницу между гибридными группами и гибридными кластерами:

1. **Гибридные группы узлов** - могут включать как развернутые в облаке узлы типа **Static**, так и серверы, развернутые в собственном ЦОД заказчика (bare-metal, виртуальные машины). Например, основную нагрузку могут нести серверы bare-metal, а облачные инстансы использоваться как масштабируемое дополнение при пиковых нагрузках. Так как и в ЦОД'е заказчика и в облаке узлы одного типа **Static**, архитектура модуля [node-manager](/modules/node-manager/) в этом случае будет соответствовать [варианту для Static-узлов](../static-nodes/). Единственным требованием является L3-связь между собственным ЦОД и облаком. Например, [Yandex.Cloud](https://yandex.cloud/) предоставляет механизм [интерконнекта](https://yandex.cloud/ru/docs/interconnect/concepts/).

2. **Гибридные кластеры** - это кластеры Deckhouse, в которых, присутствуют NodeGroup с узлами типа **Static**, развернутыми в собственном ЦОД'е заказчика (bare-metal, виртуальные машины) и NodeGroup с **CloudEphemeral**-узлами, развернутыми в облаке. В этом случае будут развернуты компоненты, необходимые для управления узлами обоих типов. Архитектура модуля [node-manager](/modules/node-manager/) будет соответствовать [варианту для Static-узлов](../static-nodes/), в дополнение будут развернуты модули, необходимые для работы **CloudEphemeral**-узлов:

    * **cloud-provider** - для взаимодействия с IaaS облака необходим установленный и настроенный облачный провайдер для этого облака. Включает также **csi-driver** и **cloud-controller-manager**.
    * **cluster-autoscaler** - обеспечивает автомасштабирования узлов кластера.

Как и для **гибридных групп**, для **гибридных кластеров** необходима L3-связь между собственным ЦОД'ом и облаком.
