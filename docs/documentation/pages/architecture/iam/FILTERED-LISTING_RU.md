В ванильном Kubernetes команда `kubectl get <resource> -A` требует кластерных
прав на операцию `list`: без них kube-apiserver возвращает `403 Forbidden`,
даже если пользователю по отдельности доступны десятки неймспейсов.

DKP расширяет kube-apiserver: пользователь без кластерных прав может явно
запросить ответ, отфильтрованный по его реальному доступу, вместо отказа.
Механизм строго опциональный — клиент, не запросивший фильтрацию, получает
байт-в-байт ванильное поведение.

## Значения --scope

| Значение | Что возвращается |
|---|---|
| `accessible` | Объекты из всех неймспейсов, где RBAC пользователя разрешает данную операцию над данным ресурсом |
| `projects` | Объекты из неймспейсов, принадлежащих любому [проекту](/modules/multitenancy-manager/cr.html#project) |
| `project:<имя>` | Объекты из неймспейсов одного конкретного проекта |
| `system` | Объекты из системных неймспейсов (не принадлежащих ни одному проекту) |

## Использование через d8

Используйте флаг `--scope` в команде `d8 k get` вместе с `-A/--all-namespaces`:

```shell
d8 k get pods -A --scope=accessible
d8 k get pods -A --scope=projects
d8 k get pods -A --scope=project:my-project
d8 k get configauditreports -A --scope=accessible   # работает и для кастомных ресурсов
d8 k get pods -A --scope=accessible -w              # watch тоже фильтруется
```

Особенности:

- Без `-A` флаг отклоняется с ошибкой: фильтровать листинг одного
  неймспейса бессмысленно.
- Если `d8 k get <resource> -A` без `--scope` завершился ошибкой
  `403 Forbidden`, CLI подсказывает подходящие варианты флага.
- Для значений флага работает shell-автодополнение, включая имена
  существующих проектов (`--scope=project:<TAB>`).

## Использование из собственного кода (HTTP API)

CLI — лишь транспорт: фильтрацию включают два HTTP-заголовка запроса к
kube-apiserver. Чтобы использовать механизм из бэкенд-сервиса, скрипта или
любого другого клиента, выставьте заголовки самостоятельно.

| Заголовок | Значение | Когда обязателен |
|---|---|---|
| `X-Deckhouse-Scope` | `accessible` \| `projects` \| `system` \| `project` | всегда (включает механизм) |
| `X-Deckhouse-Project` | имя проекта | только вместе с `X-Deckhouse-Scope: project` |

Заголовки действуют только на чтение (`list`, `get`, `watch`) основного
namespaced-ресурса без subresource, по cluster-scoped URL (путь без
`namespaces/<имя>` — тот, что формирует `-A`). Во всех остальных случаях
(мутирующие запросы, subresource, cluster-scoped ресурсы) они игнорируются.

Пример — получите список подов, доступных сервисному аккаунту:

```shell
curl -H "Authorization: Bearer $TOKEN" \
     -H "X-Deckhouse-Scope: accessible" \
     https://<kube-apiserver>/api/v1/pods
```

Пример — получите объекты Deployment одного проекта:

```shell
curl -H "Authorization: Bearer $TOKEN" \
     -H "X-Deckhouse-Scope: project" \
     -H "X-Deckhouse-Project: my-project" \
     https://<kube-apiserver>/apis/apps/v1/deployments
```

Семантика ответов:

| Ситуация | Ответ |
|---|---|
| Заголовков нет | Ванильное поведение (для пользователя без прав — `403 Forbidden`) |
| Заголовок есть, RBAC отклонил запрос | `200 OK` с объектами из доступных неймспейсов (возможно, пустой список) |
| Заголовок есть, у пользователя есть кластерные права | `200 OK`, сужение только по классификации (например, `system` отрежет проектные неймспейсы) |
| Некорректное значение заголовка у пользователя без прав | `403 Forbidden` (fail-closed) |

В client-go добавляйте заголовки через `rest.Config.WrapTransport`.

## Как это работает

1. **Фильтр авторизации kube-apiserver.** Если RBAC отклонил запрос, фильтр
   вместо немедленного `403 Forbidden` проверяет три условия: запрос — чтение
   (`list`/`get`/`watch`) основного ресурса без subresource; заголовок
   присутствует; для этого (group, resource) зарегистрирован фильтрующий
   декоратор. Только при выполнении всех трёх запрос передаётся дальше
   с пометкой «отфильтровать на уровне хранилища».

1. **Слой хранилища** вычисляет итоговое множество неймспейсов как
   пересечение двух независимых наборов:

   - **Классификация** — какие неймспейсы соответствуют значению
     `--scope`. Определяется лейблом `projects.deckhouse.io/project`,
     которым multitenancy-manager помечает неймспейсы проектов.
     Резолвится прямым запросом к kube-apiserver и от пользователя
     не зависит.
   - **Граница доступа** (в коде механизма — RBAC floor) — какие неймспейсы
     пользователю реально доступны для *данного ресурса и данной операции*.
     Вычисляется через API `authorization.deckhouse.io` (модуль user-authz):
     пересечение `AccessibleNamespaces` и `BulkSubjectAccessReview` по
     конкретной паре (resource, verb). Применяется всегда, когда RBAC
     отклонил запрос.

   В ответ попадают только объекты из неймспейсов итогового множества.

1. **Watch** (`-w`) отдаёт отфильтрованный поток и, если множество доступных
   неймспейсов меняется на лету (выдали или отозвали права, неймспейс
   сменил проект), синтезирует события `ADDED`/`DELETED` для затронутых
   объектов без переподключения клиента.

Покрыты все встроенные namespaced-ресурсы и namespaced-ресурсы CRD.

## Модель безопасности

- **Без заголовка — ванильное поведение.** Контроллеры и операторы,
  полагающиеся на `403 Forbidden` как сигнал, не ломаются.
- **Заголовок не расширяет доступ.** Граница доступа вычисляется на сервере
  из identity запроса; клиент, вручную подставивший заголовок, увидит не
  больше, чем разрешает его RBAC. Граница узкая — по конкретной паре
  (resource, verb): право `get secrets` в неймспейсе не открывает
  в нём поды.
- **Fail-closed.** Если API `authorization.deckhouse.io` недоступно
  (модуль user-authz выключен или permission-browser не отвечает),
  границу доступа вычислить нельзя — отклонённый пользователь получает
  обычный `403 Forbidden`.
- **Неразличимость отказа.** Запросы `project:<чужой>` и
  `project:<несуществующий>` дают одинаковый пустой ответ — существование
  проектов не раскрывается.

## Особый случай: namespaces

Для самого ресурса `namespaces` фильтрация применяется безусловно, без
флага и заголовков: `d8 k get namespaces` возвращает пользователю только
доступные ему неймспейсы вместо `403 Forbidden`. Фильтрация произвольных
ресурсов через `--scope` — обобщение этого же механизма, включаемое явно.

## Задействованные компоненты

| Компонент | Роль |
|---|---|
| [Deckhouse CLI](../../cli/d8/) | Флаг `--scope`, трансляция в заголовки |
| kube-apiserver (патч DKP) | Байпас `403 Forbidden` + фильтрация на слое хранилища |
| [`user-authz`](/modules/user-authz/) (permission-browser) | `AccessibleNamespaces`, `BulkSubjectAccessReview` — граница доступа |
| [`multitenancy-manager`](/modules/multitenancy-manager/) | Лейбл `projects.deckhouse.io/project` — классификация неймспейсов |
