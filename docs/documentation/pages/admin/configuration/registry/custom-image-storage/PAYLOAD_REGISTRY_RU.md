{% alert level="warning" %}
Эта страница описывает **пользовательское хранилище образов**.

Если вы хотите настроить registry для **компонентов DKP**, используйте разделы [«Кластер, управляемый DKP»](../../dkp_component/managing-interaction) или [«Managed Kubernetes: сторонний registry»](../../dkp_component/third-party.html).
{% endalert %}

DKP позволяет развернуть внутри кластера registry для образов пользовательских приложений. Для этого используйте модуль [`payload-registry`](/modules/payload-registry/).

Такой сценарий подходит, если вы хотите хранить образы рядом с кластером и не зависеть от внешнего registry в рабочих процессах приложений.

## Когда использовать payload-registry

Этот вариант подходит, если вы хотите:

- хранить образы приложений внутри кластера;
- сократить зависимость от внешнего registry;
- использовать OCI-совместимый registry для внутренних команд и сервисов;
- упростить доставку образов в изолированных или закрытых контурах.

## Что важно понимать

`payload-registry` и registry компонентов DKP — это разные сценарии.

- `payload-registry` хранит **образы приложений**;
- registry компонентов DKP хранит **образы самой платформы**.

Если нужно настроить именно образы DKP, переходите в другой раздел.

## Как включить payload-registry

1. Включите модуль `payload-registry`.

   Пример `ModuleConfig`:

   ```yaml
   apiVersion: deckhouse.io/v1alpha1
   kind: ModuleConfig
   metadata:
     name: payload-registry
   spec:
     version: 1
     enabled: true
   ```

1. Примените конфигурацию:

   ```bash
   d8 k apply -f payload-registry.yaml
   ```

1. Дождитесь, пока модуль развернёт registry и связанные ресурсы.

## Что проверить после включения

После включения модуля проверьте:

- что модуль перешёл в состояние `Ready`;
- что pod'ы registry запустились без ошибок;
- что сервис и ingress создались, если вы публикуете registry наружу;
- что пользователи или CI-системы могут пройти аутентификацию и загрузить образ.

Пример проверки модуля:

```bash
d8 k get module payload-registry -o wide
```

Пример проверки pod'ов:

```bash
d8 k get pods -A | grep payload-registry
```

## Как использовать registry

Обычно сценарий выглядит так:

1. собрать образ приложения;
1. залогиниться в registry;
1. загрузить образ;
1. использовать этот образ в манифестах приложений.

Пример команды для push зависит от того, как опубликован ваш registry и какие учётные данные вы используете.

## На что обратить внимание

### Доступ и публикация

Заранее решите, как команды и CI будут подключаться к registry:

- только изнутри кластера;
- через ingress;
- через внутреннюю сеть компании;
- через VPN.

### Сертификаты

Если registry опубликован по HTTPS, проверьте сертификаты заранее. Это поможет избежать ошибок при `docker login`, `ctr images pull` или работе CI.

### Хранение данных

Registry хранит образы и слои. Значит, ему нужно надёжное persistent storage. Перед запуском проверьте:

- какой storage class будет использоваться;
- хватит ли места под образы;
- есть ли резервное копирование, если оно нужно по вашей политике.

### Права доступа

Сразу продумайте, кто будет:

- загружать образы;
- скачивать образы;
- администрировать registry.

## Типовой сценарий использования

Например, команда собирает образ приложения в CI и публикует его во внутренний registry внутри кластера. После этого deployment использует этот образ без обращения к внешним сервисам.

Такой подход упрощает внутреннюю доставку образов и делает путь до приложения более предсказуемым.
