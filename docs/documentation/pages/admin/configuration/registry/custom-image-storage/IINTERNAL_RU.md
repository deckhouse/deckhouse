{% alert level="warning" %}
Эта страница описывает **внутренний registry для пользовательских сценариев**.

Если вы хотите настроить registry, из которого DKP скачивает свои образы, используйте раздел [«Кластер, управляемый DKP»](../../dkp_component/managing-interaction) или [«Managed Kubernetes: сторонний registry»](../../dkp_component/third-party).
{% endalert %}

{% alert level="info" %}
Если вам нужен OCI-совместимый registry для образов приложений, начните с раздела [«Payload registry»](../payload-registry). Это основной сценарий для пользовательского хранилища образов в DKP.
{% endalert %}

Внутренний registry нужен, если вы хотите хранить образы внутри контура и использовать их в приложениях, CI/CD или внутренних сервисах.

На практике под таким сценарием обычно понимают registry, который работает рядом с кластером или внутри него и обслуживает пользовательские образы.

## Когда нужен внутренний registry

Этот сценарий подходит, если вы хотите:

- хранить образы приложений внутри инфраструктуры;
- сократить зависимость от внешних registry;
- публиковать образы в закрытом контуре;
- ускорить внутренние процессы доставки образов;
- контролировать доступ к образам внутри команды или компании.

## Что важно не перепутать

В документации DKP есть два похожих сценария:

### Registry компонентов DKP

Это registry, из которого платформа скачивает **свои системные образы**.

Он нужен для работы:
- Deckhouse;
- системных модулей;
- компонентов control plane.

Для этого сценария используйте:
- [«Кластер, управляемый DKP»](../../dkp_component/managing-interaction);
- [«Managed Kubernetes: сторонний registry»](../../dkp_component/third-party).

### Внутренний registry для приложений

Это registry, который хранит **образы ваших приложений**.

Он нужен для:
- CI/CD;
- внутренних deployment-процессов;
- приложений и сервисов команд.

Для этого сценария используйте:
- [«Payload registry»](../payload-registry).

## Какой вариант обычно выбирать

Если нужен внутренний registry для приложений, чаще всего достаточно `payload-registry`.

Почему:
- модуль уже встроен в DKP;
- он разворачивает OCI-совместимый registry;
- его можно использовать как целевой registry для push и pull образов;
- он подходит для внутренних сценариев команд и сервисов.

## Что продумать заранее

Перед запуском внутреннего registry стоит определить:

### Где он будет доступен

Например:
- только внутри кластера;
- внутри корпоративной сети;
- через ingress;
- через VPN.

### Как команды будут загружать образы

Проверьте:
- как будет настроена аутентификация;
- какие учётные данные получат команды и CI;
- какие namespace, проекты или репозитории вы будете использовать.

### Где будут храниться данные

Registry хранит образы, слои и метаданные. Поэтому заранее проверьте:

- какой storage class будете использовать;
- сколько места нужно под образы;
- нужна ли репликация или резервное копирование;
- как вы будете очищать старые теги и неиспользуемые образы.

### Какие сертификаты нужны

Если registry будет доступен по HTTPS, заранее подготовьте сертификаты. Это важно для:
- `docker login`;
- `docker push`;
- `ctr images pull`;
- CI/CD-пайплайнов.

## Типовой сценарий

Обычно сценарий выглядит так:

1. команда или CI собирает образ приложения;
1. загружает его во внутренний registry;
1. deployment использует этот образ из внутреннего адреса;
1. приложение разворачивается без обращения к внешнему registry.

Такой подход помогает сделать поставку образов более предсказуемой и лучше подходит для закрытых контуров.
