В этом разделе собраны инструкции по работе с хранилищем образов контейнеров и редакциями в уже развернутом кластере Deckhouse Kubernetes Platform. Если вас интересует настройка во время установки кластера, перейдите в раздел («Установка платформы»](../../../installing/).

## Какие сценарии описаны в разделе

В разделе есть три основных сценария:
- настройка доступа DKP к хранилищу образов компонентов платформы;
- настройка пользовательского хранилища образов контейнеров;
- переключение редакции DKP.

## Хранилище образов компонентов DKP

Этот сценарий нужен, если вы хотите настроить, как DKP получает образы своих компонентов.

В кластерах, полностью управляемых DKP, для этого используется параметр `settings.registry` в `ModuleConfig` `deckhouse`. Эти настройки обрабатывает внутренний модуль `registry`.

Режим работы задается через `settings.registry.mode`. DKP поддерживает режимы `Direct`, `Proxy`, `Local` и `Unmanaged`:
- `Direct` — прямой доступ к внешнему хранилищу образов контейнеров через фиксированный внутренний адрес;
- `Proxy` — доступ через внутренний кеширующий прокси;
- `Local` — использование локального внутреннего хранилища образов компонентов DKP;
- `Unmanaged` — прямой доступ к внешнему хранилищу образов контейнеров без внутреннего механизма `registry`.

Если кластер работает в Managed Kubernetes, модуль `registry` не используется. В этом случае параметры registry меняют через `helper change-registry`.

Выберите подходящий раздел:
- [Настройки в кластере, управляемом DKP](./dkp-component/managing-interaction.html) — если кластер полностью управляется DKP.
- [Настройки в Managed Kubernetes-кластере](./dkp-component/third-party.html) — если кластер работает в Managed Kubernetes.

## Пользовательское хранилище образов контейнеров

Этот сценарий нужен, если вы хотите хранить образы пользовательских приложений внутри своей инфраструктуры.

Для этого в DKP можно использовать модуль [`payload-registry`](/modules/payload-registry/). Он реализует пользовательское хранилище образов контейнеров, соответствующее стандарту OCI.

Важно не путать этот сценарий с настройкой хранилища образов компонентов DKP:
- пользовательское хранилище используют для образов приложений, сервисов и CI/CD;
- хранилище образов компонентов DKP используют для образов самой платформы.

Подробности — в разделе [«Пользовательское хранилище»](./payload-registry.html).

## Переключение редакций

Если нужно сменить редакцию платформы, используйте раздел [«Переключение редакций»](./switching-editions.html).

## Как выбрать нужную инструкцию

Используйте эту схему:
- Нужно настроить, откуда DKP скачивает образы своих компонентов?
  - Кластер полностью управляется DKP → [Настройки в кластере, управляемом DKP](./dkp-component/managing-interaction.html)
  - Кластер работает в Managed Kubernetes → [Настройки в Managed Kubernetes-кластере](./dkp-component/third-party.html)
- Нужно отдельное хранилище образов контейнеров для пользовательских приложений? → [Пользовательское хранилище](./payload-registry.html)
- Нужно сменить редакцию DKP? → [Переключение редакций](./switching-editions.html)

## На что обратить внимание

Перед началом работ проверьте:
- какой у вас тип кластера: полностью управляемый DKP или Managed Kubernetes;
- что именно вы настраиваете: хранилище образов компонентов DKP, пользовательское хранилище образов контейнеров или редакцию платформы;
- есть ли ограничения по вашей редакции;
- готовы ли вы к возможному перезапуску компонентов, если меняете способ доступа DKP к хранилищу образов компонентов платформы.
