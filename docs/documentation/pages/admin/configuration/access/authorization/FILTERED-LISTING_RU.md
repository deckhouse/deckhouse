В Kubernetes команда `kubectl get <resource> -A` требует кластерных
прав на операцию `list`. Если кластерных прав нет, kube-apiserver возвращает ошибку `403 Forbidden`,
даже если пользователю по отдельности доступны десятки неймспейсов.

Deckhouse Kubernetes Platform (DKP) расширяет возможности kube-apiserver: пользователь без кластерных прав может явно
запросить ответ, отфильтрованный в соответствии с его фактическими правами доступа, вместо получения отказа в доступе.

Механизм является полностью опциональным.
Если клиент не запрашивает фильтрацию, поведение остаётся идентичным поведению Kubernetes.

## Использование через ⁠Deckhouse CLI

В DKP для работы с механизмом листинга с фильтрацией используется  [утилита ⁠Deckhouse CLI](../../cli/d8/).

### Области фильтрации ресурсов

С помощью флага `--scope` вы можете определить, из каких неймспейсов будут возвращаться ресурсы.

| Значение | Что возвращается |
|---|---|
| `accessible` | Объекты из всех неймспейсов, где RBAC пользователя разрешает данную операцию над данным ресурсом |
| `projects` | Объекты из неймспейсов, принадлежащих любому [проекту](/modules/multitenancy-manager/cr.html#project) |
| `project:<имя>` | Объекты из неймспейсов конкретного проекта |
| `system` | Объекты из системных неймспейсов, не принадлежащих ни одному проекту |

### Примеры использования

Используйте флаг `--scope` в команде `d8 k get` вместе с флагом `-A` (или `--all-namespaces`).

Получение объектов из всех неймспейсов, доступных пользователю:

```shell
d8 k get pods -A --scope=accessible
```

Получение объектов только из неймспейсов проектов:

```shell
d8 k get pods -A --scope=projects
```

Получение объектов указанного проекта:

```shell
d8 k get pods -A --scope=project:my-project
```

Механизм работает и для кастомных ресурсов:

```shell
d8 k get configauditreports -A --scope=accessible
```

Фильтрация также применяется к watch-запросам:

```shell
d8 k get pods -A --scope=accessible -w
```

Особенности работы:

- Использование флага `--scope` без флага `-A` отклоняется с ошибкой, поскольку фильтровать листинг в одном неймспейсе бессмысленно.
- Если команда `d8 k get <resource> -A` без указания `--scope` завершается ошибкой `403 Forbidden`, ⁠Deckhouse CLI подсказывает подходящие варианты использования флага.
- Для значений флага `--scope` работает автоматическое дополнение в командной оболочке (shell completion), включая имена существующих проектов (`--scope=project:<TAB>`).

## Использование из собственного кода (HTTP API)

CLI является лишь транспортным механизмом. Фильтрацию включают два HTTP-заголовка запроса к
kube-apiserver. Чтобы использовать механизм из собственного сервиса, скрипта или
любого другого клиента, передавайте заголовки самостоятельно.

| Заголовок | Значение | Когда обязателен |
|---|---|---|
| `X-Deckhouse-Scope` | `accessible` \| `projects` \| `system` \| `project` | Всегда (включает механизм) |
| `X-Deckhouse-Project` | Имя проекта | Только при использовании `X-Deckhouse-Scope: project` |

Заголовки применяются только к операциям чтения (`list`, `get`, `watch`) для namespaced-ресурсов без субресурсов, выполняемым через cluster-scoped URL-адрес (то есть по пути без
`namespaces/<имя>`, который формируется при использовании флага `-A`).

Во всех остальных случаях
(мутирующие запросы, обращения к субресурсам и cluster-scoped-ресурсам) заголовки игнорируются.

Пример получения списка подов, доступных сервисному аккаунту:

```shell
curl -H "Authorization: Bearer $TOKEN" \
     -H "X-Deckhouse-Scope: accessible" \
     https://<kube-apiserver>/api/v1/pods
```

Пример получения списка объектов Deployment одного проекта:

```shell
curl -H "Authorization: Bearer $TOKEN" \
     -H "X-Deckhouse-Scope: project" \
     -H "X-Deckhouse-Project: my-project" \
     https://<kube-apiserver>/apis/apps/v1/deployments
```

Семантика ответов:

| Ситуация | Ответ |
|---|---|
| Заголовки отсутствуют | Стандартное поведение Kubernetes (ошибка `403 Forbidden` для пользователя без необходимых прав) |
| Заголовок присутствует, но RBAC отклонил запрос | Ответ `200 OK` со списком объектов из доступных неймспейсов (список может быть пустым) |
| Заголовок присутствует, и у пользователя есть кластерные права | Ответ `200 OK`. Фильтрация применяется только по классификации неймспейсов (например, `system` исключает проектные неймспейсы) |
| У пользователя нет кластерных прав и передано некорректное значение заголовка | Ошибка `403 Forbidden` (fail-closed) |

При использовании client-go добавляйте заголовки через `rest.Config.WrapTransport`.

## Принцип работы

1. **Фильтр авторизации kube-apiserver.**

   Если RBAC отклоняет запрос, фильтр
   вместо немедленного возврата ошибки `403 Forbidden` проверяет три условия:

   - выполняется операция чтения (`list`, `get` или `watch`);
   - запрос направлен к основному ресурсу, а не субресурсу;
   - для данного ресурса зарегистрирован фильтрующий
   декоратор и присутствует заголовок.

   Только при выполнении всех трёх условий запрос передаётся дальше с пометкой о необходимости фильтрации на уровне хранилища.

1. **Слой хранилища** вычисляет итоговое множество неймспейсов как
   пересечение двух независимых наборов:

   - **Классификация** — какие неймспейсы соответствуют значению
     `--scope`. Определяется лейблом `projects.deckhouse.io/project`,
     которым [модуль `multitenancy-manager`](/modules/multitenancy-manager/) помечает неймспейсы проектов.
     Резолвится прямым запросом к kube-apiserver и не зависит от пользователя.
   - **Граница доступа** (в коде механизма — RBAC floor) — какие неймспейсы
     пользователю реально доступны для *данного ресурса и данной операции*.
     Вычисляется через API `authorization.deckhouse.io` ([модуль `user-authz`](/modules/user-authz/)) как
     пересечение результатов `AccessibleNamespaces` и `BulkSubjectAccessReview` по
     конкретной паре (`resource`, `verb`). Применяется всегда, когда RBAC
     отклонил запрос.

   В ответ попадают только объекты из неймспейсов итогового множества.

1. **Watch** (`-w`) — отдаёт отфильтрованный поток и, если множество доступных
   неймспейсов меняется «на лету» (например, пользователю выдали или отозвали права или неймспейс
   был перенесён в другой проект), синтезирует события `ADDED`/`DELETED` для затронутых
   объектов без повторного подключения клиента.

Механизм поддерживает все встроенные namespaced-ресурсы Kubernetes, а также namespaced-ресурсы, определённые через CRD.

## Модель безопасности

- **Без заголовков используется стандартное поведение Kubernetes.**
  Контроллеры и операторы, полагающиеся на ошибку `403 Forbidden` как сигнал, продолжают работать без изменений.
- **Заголовки не расширяет права доступа.**
  Граница доступа вычисляется на сервере из основании идентичности пользователя, выполнившего запрос.
  Даже при ручной подстановке заголовков пользователь не сможет получить доступ к объектам, запрещённым его RBAC-правами.
  Проверка выполняется отдельно для каждой пары
  (`resource`, `verb`). Таким образом, например, наличие права `get secrets` в неймспейсе не предоставляет доступ к подам в нём.
- **Используется модель fail-closed.** Если API `authorization.deckhouse.io` недоступен
  (например, если модуль `user-authz` выключен или сервис `permission-browser` не отвечает),
  границу доступа вычислить нельзя — отклонённый пользователь получает
  стандартный ответ `403 Forbidden`.
- **Неразличимость отказа.** Запросы вида `project:<чужой>` и
  `project:<несуществующий>` дают одинаковый пустой ответ — существование
  проектов не раскрывается.

## Особый случай: namespaces

Для ресурса `namespaces` фильтрация применяется безусловно, независимо от использования флагов и заголовков.
Команда `d8 k get namespaces` возвращает пользователю только
доступные ему неймспейсы вместо ошибки `403 Forbidden`. Фильтрация произвольных
ресурсов через `--scope` является обобщением этого механизма, которое включается явно.

## Задействованные компоненты

| Компонент | Роль |
|---|---|
| [Deckhouse CLI](../../cli/d8/) | Поддержка флага `--scope` и преобразование его значений в заголовки |
| kube-apiserver (патч DKP) | Перехват `403 Forbidden` и фильтрация на слое хранилища |
| Модуль [`user-authz`](/modules/user-authz/) (`permission-browser`) | Определение границы доступа через `AccessibleNamespaces` и `BulkSubjectAccessReview` |
| Модуль [`multitenancy-manager`](/modules/multitenancy-manager/) | Классификация неймспейсов по лейблу `projects.deckhouse.io/project` |
