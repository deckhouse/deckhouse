## API

_API Deckhouse Kubernetes Platform_ — это интерфейс взаимодействия пользователей и компонентов DKP с кластером. Все операции в DKP выполняются через API, что обеспечивает единообразие и стандартизацию взаимодействия. API Deckhouse Kubernetes Platform полностью совместим с API Kubernetes.

### CustomResouseDefinitions

_CustomResouseDefinitions (CRD)_ — механизм расширения API, позволяющий создавать новые типы ресурсов в кластере. CRD помогают обеспечить гибкость и расширяемость Deckhouse Kubernetes Platform.

## Кластер

_Кластер_ — группа взаимосвязанных узлов, управляемая Deckhouse Kubernetes Platform и обеспечивающая единую среду для развёртывания, масштабирования и обслуживания [полезной нагрузки](#полезная-нагрузка).

Кластер состоит из [системы](#система), обеспечивающей ее функционирование, и [полезной нагрузки](#полезная-нагрузка), которая запускается в кластере.

## Система

_Система_ — часть кластера, обеспечивающая его работоспособность, управление ресурсами и автоматизацию процессов. Включает в себя подсистемы, в свою очередь состоящие из [модулей](#модуль) и [системных компонентов](#системный-компонент). Система работает преимущественно в [системных пространствах имён](#системное-пространство-имён), но некоторые её части (обслуживающие нагрузку конкретного проекта) могут работать в [пользовательских пространствах имён](#пользовательское-пространство-имён).

### Модуль

_Модуль_ — набор ресурсов и приложений, предназначенных для расширения функциональности Deckhouse Kubernetes Platform. Каждый модуль выполняет определённую роль (например, сбор метрик, балансировка трафика, управление секретами).
Документация по модулю и его [API](#api) доступна в [библиотеке модулей](/modules/).

### Системный компонент

_Системный компонент_ — единичный сервис или контроллер внутри системы, выполняющий конкретную роль. Системные компоненты обмениваются данными между собой и обеспечивают работу [системы](#система).

## Полезная нагрузка

_Полезная нагрузка (workload)_ — совокупность всех [проектов](#проекты), включая используемые ими дисковые, сетевые и другие ресурсы.

## Пространство имён

_Пространство имён (namespace)_ — механизм логической группировки части объектов (нагрузки) в Kubernetes.

Пример объектов Kubernetes, которые могут быть сгруппированы с использованием пространства имён: поды и их контроллеры (Deployment, ReplicaSet, StatefulSet, DaemonSet, Job/CronJob), сервисы (Service), ConfigMap, Secret, PersistentVolumeClaim и другие.

Пространства имён в Kubernetes часто применяются для изоляции объектов, хотя, без дополнительной настройки, они этого не обеспечивают, в отличие от [проектов](#проекты) в Deckhouse Kubernetes Platform.
 
Пространства имен, создаваемые Deckhouse Kubernetes Platform, разделяются на [системные](#системное-пространство-имён) и [пользовательские](#пользовательское-пространство-имён). 

### Системное пространство имён

_Системное пространство имён_ — [пространство имён](#пространство-имён), в котором запускается нагрузка, обеспечивающая работоспособность [системы](#система).

Системные пространства имен, используемые в Deckhouse Kubernetes Platform, имеют следующие префиксы:
- `d8-` 
- `kube-`

### Пользовательское пространство имён

_Пользовательское пространство имён_ — [пространство имён](#пространство-имён), в котором запускается [полезная нагрузка](#полезная-нагрузка).

Пользовательское пространство имен всегда относится к одному [проекту](#проекты) и не может иметь префикс [системного пространства имен](#системное-пространство-имён).

> РЕМАРКА: Системные пространства имен же тоже образуют проекты, системные. Вариант - убрать отсылку к проектам здесь. Либо добавить в системных пространствах имён, что они образуют системные проекты. Но тогда надо дать определение системного проекта.

## Проекты

_Проекты_ — механизм логического разделения кластера Deckhouse Kubernetes Platform на части. Проект — основная единица изоляции и управления в Deckhouse Kubernetes Platform.

Проекты обладают следующими преимуществами перед пространствами имен Kubernetes:
- **Делегирование управления** — возможность пользователям и командам самостоятельно работать с частью кластера (развертывать приложения, настраивать ресурсы, управлять доступом других пользователей и команд), без прямого участия администратора кластера.
- **Изоляция ресурсов** — проекты обеспечивают действительно изолированное окружение, где разработчики могут развернуть экземпляр приложения без влияния на другие проекты. Проекты позволяют разделить приложение на логические части и назначать для каждого проектного или дополнительного проектного пространства имён отдельные политики доступа.
- **Шаблонизация** — создание проектов по шаблону, для избежания ошибок в конфигурации проектов и обеспечения единообразия. При изменении шаблона проекта, все связанные с ним проекты будут обновлены в соответствии с новым шаблоном.
- **Управление доступностью функций** — администратор используя [ролевую модель](#ролевая-модель) может управлять тем, какие возможности Deckhouse Kubernetes Platform доступны в проекте. Например, запретить в проекте использовать возможность запуска виртуальных машин. Или наоборот, разрешить запускать виртуальные машины, но запретить развертывание приложений в контейнерах.
 
Deckhouse Kubernetes Platform использует пространства имён при работе с [проектами](#проекты), обеспечивая совместимость с пространствами имен кластера Kubernetes. Это позволяет работать пользовательской нагрузке как в рамках кластера Deckhouse, так и без Deckhouse (в случае его удаления), в рамках кластера Kubernetes.

### Основное проектное пространство имён

_Основное проектное пространство имён_ — основное [пространство имён](#пространство-имён) внутри [проекта](#проекты). Используется как для управления самим проектом (создания [дополнительных проектных пространств имён](#дополнительное-проектное-пространство-имён), управления правами), так и для запуска [полезной нагрузки](#полезная-нагрузка).

Основное проектное пространство имён может содержать несколько [дополнительных проектных пространств имён](#дополнительное-проектное-пространство-имён), которые позволяют логически разделить сервисы и ресурсы внутри проекта, а также применить к ним отдельные политики доступа и лимиты ресурсов.

### Дополнительное проектное пространство имён

_Дополнительное проектное пространство имён_ — [пространство имён](#пространство-имён), которое создается внутри [проекта](#проекты), как и [основное проектное пространство имен](#основное-проектное-пространство-имён). Дополнительное проектное пространство имён позволяет логически разделить полезную нагрузку внутри проекта, а также применять к ним отдельные политики доступа. 

Дополнительные проектные пространства имён не могут обладать доступами, которых нет у [основного проектного пространства имён](#основное-проектное-пространство-имён).

## Ролевая модель

_Ролевая модель_ — механизм управления доступом на основе ролей, который определяет, какие пользователи (или сервисы) могут выполнять конкретные действия с ресурсами в рамках [проектов](#проекты) или [системы](#система).

В Deckhouse Kubernetes Platform реализовано две ролевые модели:
- **Системная ролевая модель** — используется для управления доступом к [системе](#система) или ее частям.
  Уровни доступа: <-- **надо уточнить !**
  - просмотр (viewer)
  - изменение (manager)
  - управление (superadmin)
- **Проектная ролевая модель** — используется для управления доступом к [полезной нагрузке](#полезная-нагрузка).

  Область действия:
  - вся полезная нагрузка
  - проект
  - проектное пространство имён

  Уровни доступа:
    - просмотр (viewer)
    - использование (user)
    - изменение (manager)
    - управление (admin)
    - неограниченный доступ (superadmin) 

  Администратор может создавать другие проектные роли или изменять существующие.
