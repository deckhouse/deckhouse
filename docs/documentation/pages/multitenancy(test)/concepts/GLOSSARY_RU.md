На этой странице собраны термины и определения, используемые в документации Deckhouse Kubernetes Platform.

## Deckhouse

_Deckhouse (экосистема Deckhouse)_ — совокупность взаимосвязанных продуктов, обеспечивающая управление кластерами Kubernetes, разработку, поддержку и эксплуатацию ПО. Экосистема Deckhouse включает такие продукты, как:
- [Deckhouse Kubernetes Platform](https://deckhouse.ru/products/kubernetes-platform/) — платформа управления контейнерными нагрузками
- [Deckhouse Virtualization Platform](https://deckhouse.ru/products/virtualization-platform/) — платформа управления облачной инфраструктурой
- [Deckhouse Observability Platform](https://deckhouse.ru/products/observability-platform/) — платформа централизованного мониторинга и журналирования инфраструктуры и приложений
- [Deckhouse Commander](https://deckhouse.ru/products/commander/) — решение для управления кластерами Deckhouse Kubernetes Platform
- [Deckhouse Stronghold](https://deckhouse.ru/products/stronghold/) — решение для безопасного хранения и управления секретами

Подробнее об экосистеме Deckhouse и продуктах можно узнать на [официальном сайте](https://deckhouse.ru/).

## CustomResouseDefinitions

_CustomResouseDefinitions (CRD)_ — описание ресурсов в [кластере](#кластер), которое позволяет расширять его функциональность. CRD определяют состав параметров и правила их проверки (схему). После установки CRD в кластере, появляется возможность создавать объекты, соответствующие схеме CRD.

## Кластер

_Кластер (Cluster)_ — группа взаимосвязанных узлов, управляемая Deckhouse Kubernetes Platform и обеспечивающая единую среду для развёртывания, масштабирования и обслуживания [полезной нагрузки](#полезная-нагрузка).

Основные компоненты кластера:
- [Система](#система)
- [Проекты](#проект)

## Система

_Система (System)_ — часть кластера, обеспечивающая его работоспособность, управление ресурсами и автоматизацию процессов. Включает в себя подсистемы, в свою очередь состоящие из [модулей](#модуль) и [системных компонентов](#системный-компонент). Система работает в выделенных [системных пространствах имён](#системное-пространство-имён).

## Модуль

_Модуль (Module)_ — набор ресурсов и приложений, предназначенных для расширения функциональности Deckhouse Kubernetes Platform. Каждый модуль выполняет определённую роль (например, сбор метрик, балансировка трафика, управление секретами). Модуль настраивается с помощью ресурса ModuleConfig с именем модуля в kebab case (например, `vertical-pod-autoscaler`) и может устанавливать [CustomResouseDefinitions (CRD)](#customresousedefinitions), которые также используются для настройки работы модуля.

Документация по модулю, его настройкам и списке его CRD доступна в [библиотеке модулей](/modules/).

## Системный компонент

_Системный компонент (System Component)_ — единичный сервис или контроллер внутри системы, выполняющий конкретную роль. Системные компоненты обмениваются данными между собой и обеспечивают работу [системы](#система).

## Полезная нагрузка

_Полезная нагрузка (Workload)_ — совокупность всех пользовательских приложений, запущенных в рамках [проектов](#проект), включая используемые ими дисковые, сетевые и другие ресурсы.

## Пространство имён

_Пространство имён (Namespace)_ — механизм логической изоляции ресурсов в [кластере](#кластер). Deckhouse Kubernetes Platform использует пространства имён при работе с [проектами](#проекты), обеспечивая совместимость с пространствами имен кластера Kubernetes. Это позволяет работать пользовательской нагрузке как в рамках кластера Deckhouse, так и без Deckhouse (в случае его удаления), в рамках кластера Kubernetes. 

Пространства имен, создаваемые Deckhouse, разделяются на [системные](#системное-пространство-имён) и [пользовательские](#пользовательское-пространство-имён).

### Системное пространство имён

_Системное пространство имён_ — логическая группа ресурсов, обеспечивающая работоспособность [системы](#система).

Системные пространства имен:
- `d8-*` 
- `kube-*`

### Пользовательское пространство имён

_Пользовательское пространство имён_ — логическая группа ресурсов, расположенных внутри [кластера](#кластер), предназначенная для работы с [полезной нагрузкой](#полезная-нагрузка). Пользовательские пространства управляются Deckhouse при работе c [проектами](#проект).

## Проекты

_Проекты_ — механизм улучшенной логической изоляции ресурсов в [кластере](#кластер) Deckhouse Kubernetes Platform. Проекты используются Deckhouse Kubernetes Platform для управления ресурсами кластера на уровне приложений, пользователей и независимых команд. Проекты представляют собой более высокоуровневый механизм по сравнению с классической изоляцией на уровне пространств имен Kubernetes. 

Кроме изоляции ресурсов, проекты позволяют:
- Управлять доступностью функций Deckhouse Kubernetes Platform в рамках проекта.
- Настраивать безопасность (ролевая модель, политики безопасности).
- Контролировать квоты на использование сетевых ресурсов и систем хранения.
- Настраивать изоляцию ресурсов внутри проектов с помощью [дополнительных проектных пространств имён](#дополнительное-проектное-пространство-имён).
- Расширять функциональность под свои нужды.

Проекты в Deckhouse Kubernetes Platform сочетают стандартные возможности Kubernetes с дополнительными абстракциями, упрощающими использование кластера приложениями, пользователями или командами.

### Проект

_Проект (Project)_ — основная единица изоляции и управления ресурсами в [проектах](#проекты) Deckhouse Kubernetes Platform.

> РЕМАРКА: тут может добавить про ресурсы, которыми управляется проект. Но для глоссария может быть перебор. 

### Основное проектное пространство имён

_Основное проектное пространство имён_ — основное [пространство имён](#пространство-имён) внутри [проекта](#проект). Используется как для управления самим проектом (создания [дополнительных проектных пространств имён](#дополнительное-проектное-пространство-имён), управления правами), так и для запуска [полезной нагрузки](#полезная-нагрузка).

### Дополнительное проектное пространство имён

_Дополнительное проектное пространство имён_ — создаются внутри [проекта](#проект) и позволяют логически разделить сервисы и ресурсы, а также применить к ним отдельные политики доступа. Управляется ресурсом ProjectNamespace.

## Ролевая модель

_Ролевая модель (RBAC)_ — механизм управления доступом на основе [ролей](#роль), который определяет, какие пользователи (или сервисы) могут выполнять конкретные действия с ресурсами в рамках [проектов](#проект) или [кластера](#кластер).

### Роль

_Роль (Role)_ — набор прав на выполнение действий с ресурсами в рамках [кластера](#кластер) или [проекта](#проект).

Типы ролей:
- _Кластерная роль_ — определяет доступ в рамках [кластера](#кластер).
- _Проектная роль_ — определяет доступ в рамках [проекта](#проект).
