При планировании размещения и запуска на одном узле более чем 100 подов, к узлу предъявляются дополнительные требования. Особенные требования затрагивают диск (распаковка и монтирование слоёв образа), ядро и память узла. Приведённые ниже рекомендации помогут получить более предсказуемое время старта подов или ускорить его.

{% alert %}
Для узлов с высокой плотностью подов:

- заранее спланируйте размер подсети узла ([podSubnetNodeCIDRPrefix](/products/kubernetes-platform/documentation/v1/reference/api/cr.html#clusterconfiguration-podsubnetnodecidrprefix)) — лимит подов на узел подстроится автоматически;
- используйте ядро Linux версии 6.12 или новее;
- используйте быстрый локальный диск (NVMe/SSD), не используйте сетевую файловую систему;
- готовьте лёгкие образы приложений (минимум слоёв, например`distroless`).
{% endalert %}

Компоненты Control plane для подобных сценариев Deckhouse настраивает автоматически — дополнительная ручная настройка не требуется.

## Лимит подов на узел

{% alert %}
Лимит подов на узел Deckhouse вычисляется на основе размера подсети узла — параметра [podSubnetNodeCIDRPrefix](/products/kubernetes-platform/documentation/v1/reference/api/cr.html#clusterconfiguration-podsubnetnodecidrprefix). Чтобы узел вмещал до 1000 подов, выберите при развёртывании кластера `podSubnetNodeCIDRPrefix` ≤ `21`.
{% endalert %}

Размер подсети, выделяемой узлу, определяет лимит подов на узел:

| `podSubnetNodeCIDRPrefix` | Подов на узел (по умолчанию) |
|---|---|
| ≥ `24` | 120 |
| `23` | 250 |
| `22` | 500 |
| ≤ `21` | 1000 |

Параметр `podSubnetNodeCIDRPrefix` задаётся при развёртывании кластера, поэтому если вы заранее знаете о планах на высокую плотность подов, выберите подходящее значение сразу.

## Ядро узла

{% alert %}
Для узлов с высоким количеством подов рекомендуется ядро Linux версии 6.12 или новее с поддержкой модуля `erofs`.
{% endalert %}

Начиная с версии `2.x`, `containerd` поддерживает использование функциональности `EROFS`, зависящей от опции ядра `CONFIG_EROFS_FS_BACKED_BY_FILE`. Эта опция включена по умолчанию начиная с Linux 6.12 и позволяет напрямую монтировать слои образов, представленные обычными файлами, без создания отдельного loop-устройства для каждого слоя или контейнера.

При отсутствии этой опции запуск большого количества контейнеров может приводить к созданию большого числа loop-устройств. На узлах с сотнями или тысячами подов это увеличивает нагрузку на систему и может существенно замедлять запуск контейнеров.

Поэтому рекомендуется для узлов, на которых планируется запуск большого количества подов, выбирать образ операционной системы с ядром 6.12 или новее.

## Диск узла

{% alert %}
Используйте быстрый локальный диск (NVMe или SSD). Не размещайте файлы контейнерной среды и kubelet на сетевой файловой системе.
{% endalert %}

Запуск каждого пода включает распаковку и работу со слоями образа — это дискозависимые операции. При массовом запуске эти операции умножаются на количество подов, поэтому диск напрямую определяет время старта.

- Базовая рекомендация для узлов — быстрые диски с производительностью не менее 400 операций ввода-вывода в секунду (подробнее в [«Подготовка к production»](./production.html#требования-к-ресурсам)). Для узлов с высокой плотностью подов выбирайте заметно более производительные диски — предпочтительно локальные NVMe.
- Не используйте сетевую или распределённую файловую систему под каталоги контейнерной среды и kubelet: сетевые задержки на операциях монтирования умножаются на количество подов и делают время старта непредсказуемым.

## Образы приложений

{% alert %}
Чем меньше и эффективнее собран образ, тем быстрее стартует под. Используйте минимальные образы (например, `distroless`) и сокращайте количество слоёв.
{% endalert %}

Время на скачивание образа и дисковая нагрузка при его старте зависят от размера и структуры образа. При большом количестве подов это может стать проблемой:

- используйте минимальные базовые образы (например, `distroless` или `*-slim`/`alpine`), без лишних пакетов и инструментов;
- сокращайте количество слоёв образа и при возможности объединяйте их, чтобы уменьшить объём распаковки и монтирования;
- держите образы небольшими — удаляйте кеши пакетных менеджеров, артефакты сборки и временные файлы;
- используйте многоэтапную сборку.

Также причиной задержки старта подов может быть не сам образ, а хранилище образов контейнеров и канал до него: при массовом запуске (обновлении) множества приложений узел может одновременно скачивать несколько образов, тогда хранилище вместе с сетью до него тоже может стать узким местом. Рекомендуется:

- размещать хранилище образов контейнеров в сетевой близости к кластеру и обеспечивать достаточную пропускную способность канала до него;
- для образов из внешних (в том числе публичных) хранилищ образов контейнеров используйте локальное зеркало или кеш с автоматической подгрузкой рядом с кластером.

## Ресурсы узла

Помимо диска и ядра, плотность подов ограничена процессором и памятью узла. На примере тестовой нагрузки (под — статический Nginx):

- узел 4 процессора / 8 ГБ оперативной памяти с SSD уверенно держит одновременный скейлинг с нуля в 150 подов;
- узел 32 процессора / 64 ГБ оперативной памяти с локальным NVMe и ядром ≥ 6.12 запускает порядка 1000 подов за несколько минут.

Реальные значения зависят от характера приложений, поэтому перед выходом в production-окружение рекомендуется провести нагрузочное тестирование.
