Внешние проверки статуса позволяют администраторам проекта отправлять данные запроса на слияние во внешний сервис и использовать ответ сервиса как дополнительное условие слияния. Используйте их, если перед слиянием запрос должен пройти внешнюю проверку соответствия требованиям, контроль качества или проверку безопасности.

Внешние проверки статуса настраиваются отдельно для каждого проекта и не наследуются другими проектами.

## Доступ к настройкам

Чтобы настроить внешние проверки статуса, откройте проект и перейдите в **Settings** → **Merge requests**.

На странице используются две связанные области настроек:

- **Merge checks** — настройки проекта, влияющие на возможность слияния.
- **External status checks** — таблица для создания, обновления и удаления внешних сервисов проверки статуса.

Пользователю нужно право на управление настройками запросов на слияние в проекте. В стандартных ролях оно доступно пользователям с ролью **Maintainer** или **Owner**.

## Настройки Merge checks

### Status checks must succeed

Флажок **Status checks must succeed** блокирует запросы на слияние до тех пор, пока все применимые внешние проверки статуса не перейдут в состояние `passed`.

Если флажок выключен, внешние проверки статуса продолжают отображаться в запросах на слияние, но не блокируют слияние.

### Status checks timeout

Поле **Status checks timeout** задает таймаут по умолчанию для проверок статуса в проекте.

Поведение:

- Значение по умолчанию — `5` минут.
- Значение должно быть `1` минута или больше.
- Значение применяется к проверкам без индивидуального таймаута.
- Если внешний сервис не ответил до истечения таймаута, ответ проверки переходит в состояние `failed`.

## Таблица External status checks

Таблица **External status checks** показывает все сервисы проверки статуса, настроенные в проекте.

В таблице отображаются:

- Название проверки статуса.
- URL внешнего сервиса.
- Область применения по целевым веткам.
- Таймаут проверки или таймаут проекта по умолчанию.
- Состояние HMAC shared secret.
- Действия **Update** и **Delete**.

Если у проверки не задан индивидуальный таймаут, в таблице отображается таймаут проекта с пометкой `(default)`.

## Добавление внешней проверки статуса

Чтобы добавить внешнюю проверку статуса:

1. Откройте проект.
1. Перейдите в **Settings** → **Merge requests**.
1. В блоке **External status checks** нажмите **Add external status check**.
1. Заполните поля.
1. Нажмите **New external status check**.

| Поле | Описание |
|------|----------|
| **Service name** | Название внешнего сервиса. Значение обязательно, должно быть уникальным в проекте и не должно превышать 255 символов. |
| **API to check** | URL endpoint'а внешнего сервиса. Значение обязательно, должно быть уникальным в проекте и должно использовать протокол `http` или `https`. |
| **Target branch** | Область, определяющая целевые ветки запросов на слияние, для которых применяется проверка. |
| **Timeout minutes** | Индивидуальный таймаут проверки. Если задан, переопределяет значение проекта **Status checks timeout**. |
| **HMAC Shared Secret** | Необязательный секрет для подписи запросов, отправляемых из Deckhouse Code во внешний сервис. |

После создания проверки Deckhouse Code создает ответы в состоянии `pending` для подходящих открытых запросов на слияние. Запросы во внешний сервис для таких уже существующих запросов на слияние не отправляются. Эти ответы перейдут в состояние `failed` после истечения таймаута, если пользователь не перезапустит проверку.

Для новых событий запросов на слияние Deckhouse Code автоматически отправляет запросы в подходящие внешние сервисы проверки статуса.

## Область применения по целевым веткам

Поле **Target branch** определяет, для каких запросов на слияние используется проверка статуса.

| Область | Описание |
|---------|----------|
| **All branches** | Проверка применяется к запросам на слияние в любую целевую ветку. |
| **All protected branches** | Проверка применяется к запросам на слияние в защищенные ветки. |
| Выбранные protected branches | Проверка применяется только к запросам на слияние в выбранные защищенные ветки. Поддерживаются wildcard protected branches. |

Когда целевая ветка запроса на слияние меняется, Deckhouse Code пересчитывает применимые проверки. Ответы для проверок, которые больше не применяются, удаляются, а для новых применимых проверок создаются недостающие ответы в состоянии `pending`.

## HMAC shared secret

Если задан **HMAC Shared Secret**, Deckhouse Code добавляет заголовок `X-Gitlab-Signature` к запросам во внешний сервис.

Значение заголовка — HMAC-SHA256 от тела запроса, сформированный с использованием shared secret.

Секрет хранится в зашифрованном виде. После сохранения UI показывает только факт наличия секрета. Чтобы заменить секрет:

1. В строке проверки нажмите **Update**.
1. Нажмите **Edit secret**.
1. Введите новое значение.
1. Нажмите **Update status check**.

## Жизненный цикл проверки

Когда происходит событие запроса на слияние, Deckhouse Code отправляет payload запроса на слияние во все применимые внешние сервисы проверки статуса. Payload содержит объект `external_approval_rule` с `id`, `name` и `external_url` проверки.

Ответ проверки может иметь один из следующих статусов:

| Статус | Описание |
|--------|----------|
| `pending` | Deckhouse Code ожидает ответ внешнего сервиса. |
| `passed` | Внешний сервис подтвердил состояние запроса на слияние. |
| `failed` | Внешний сервис отклонил состояние запроса на слияние, запрос к внешнему URL завершился ошибкой или истек таймаут. |

Внешний сервис обновляет ответ через API endpoint:

```text
POST /projects/:id/merge_requests/:merge_request_iid/status_check_responses
```

Запрос должен содержать:

- `external_status_check_id` — ID проверки статуса.
- `sha` — текущий `HEAD` SHA source branch запроса на слияние.
- `status` — `passed` или `failed`.

Deckhouse Code не обновляет ответ, если:

- `sha` не совпадает с текущим `HEAD` source branch.
- Ответ больше не находится в состоянии `pending`.
- Таймаут уже истек.

## Таймауты

Отсчет таймаута начинается в момент отправки запроса во внешний сервис. Если пользователь перезапускает failed-проверку, отсчет таймаута начинается с момента перезапуска.

Значение таймаута выбирается в следующем порядке:

1. Значение **Timeout minutes** у проверки статуса.
1. Значение проекта **Status checks timeout**.

Фоновый worker проверяет ответы в состоянии `pending` каждую минуту. Когда ответ превышает свой таймаут, Deckhouse Code переводит его в состояние `failed` и записывает причину `Automatically closed after timeout`.

Если запрос во внешний сервис завершился ошибкой, Deckhouse Code переводит ответ в состояние `failed` и сохраняет причину ошибки. Пользователи могут увидеть причину ошибки в виджете запроса на слияние.

## Удаление внешней проверки статуса

Чтобы удалить проверку:

1. Откройте проект.
1. Перейдите в **Settings** → **Merge requests**.
1. В блоке **External status checks** нажмите **Delete** для нужной проверки.
1. Подтвердите удаление.

После удаления проверка больше не применяется к запросам на слияние проекта.

## События аудита

Deckhouse Code записывает события аудита для управления проверками статуса и изменения ответов.

| Событие аудита | Описание |
|----------------|----------|
| `external_status_check_created` | Создана внешняя проверка статуса. |
| `external_status_check_updated` | Обновлена внешняя проверка статуса. |
| `external_status_check_destroyed` | Удалена внешняя проверка статуса. |
| `external_status_check_response_updated` | Ответ изменен через callback внешнего сервиса, перезапуск, ошибку запроса или таймаут. |

## Права доступа

| Действие | Минимальная роль |
|----------|------------------|
| Создать, обновить, удалить или посмотреть сервисы проверки статуса проекта | **Maintainer** или **Owner** |
| Отправить callback со статусом проверки | **Developer** |
| Смотреть ответы проверок статуса в виджете запроса на слияние | **Reporter** |
| Смотреть URL внешней проверки статуса в виджете запроса на слияние | **Developer** или auditor |
| Перезапустить failed-проверку | **Developer** |

Для internal projects аутентифицированные пользователи, которые могут читать запросы на слияние в целевом проекте, также могут читать ответы проверок статуса.

## Troubleshooting

### Ошибка дублирования имени или URL

В проекте не может быть двух внешних проверок статуса с одинаковым **Service name** или **API to check**. Используйте уникальные значения для каждой проверки.
