Режим обслуживания позволяет администраторам свести операции записи к минимуму на время проведения регламентных работ.
Основная цель — заблокировать все внешние действия, изменяющие внутреннее состояние.
Внутреннее состояние включает базу данных PostgreSQL, а также файлы, Git-репозитории и репозитории контейнеров.

Когда режим обслуживания включён, выполняющиеся действия завершаются относительно быстро, потому что новые действия не поступают,
а изменения внутреннего состояния минимальны. В таком состоянии проще выполнять различные регламентные работы.

Режим обслуживания разрешает большинство внешних действий, которые не изменяют внутреннее состояние.
На верхнем уровне HTTP-запросы `POST`, `PUT`, `PATCH` и `DELETE` блокируются.

## Включение режима обслуживания

Включите режим обслуживания от имени администратора одним из способов:

- **Веб-интерфейс**:
  1. Перейдите в «Admin» → «Настройки» → «Основные».
  1. Разверните раздел «Режим обслуживания» и установите флажок «Включить режим обслуживания». При необходимости можно также добавить сообщение для баннера.
  1. Нажмите «Сохранить изменения».
- **API**:

  ```shell
  curl --request PUT --header "PRIVATE-TOKEN: $ADMIN_TOKEN" \
    "<instance-url>/api/v4/application/settings?maintenance_mode=true"
  ```

## Отключение режима обслуживания

Отключите режим обслуживания одним из способов:

- **Веб-интерфейс**:
  1. Перейдите в «Admin» → «Настройки» → «Основные».
  1. Разверните раздел «Режим обслуживания» и снимите флажок «Включить режим обслуживания».
  1. Нажмите «Сохранить изменения».
- **API**:

  ```shell
  curl --request PUT --header "PRIVATE-TOKEN: $ADMIN_TOKEN" \
    "<instance-url>/api/v4/application/settings?maintenance_mode=false"
  ```

## Поведение функций в режиме обслуживания

Когда режим обслуживания включён, вверху страницы отображается баннер.
Баннер можно настроить с собственным сообщением.

При попытке пользователя выполнить недопустимую операцию записи отображается ошибка.

{% alert level="info" %}
В некоторых случаях визуальная реакция на действие может вводить в заблуждение.
Например, при добавлении проекта в избранное кнопка «В избранное» меняется на действие «Убрать из избранного».
Однако это лишь обновляет интерфейс и не учитывает статус `POST`-запроса.
{% endalert %}

### Функции администратора

Администраторы могут редактировать настройки приложения. Это позволяет отключить режим обслуживания после его включения.

### Аутентификация

Все пользователи могут входить в систему и выходить из неё, но новые пользователи не создаются.

Если на это время запланированы синхронизации LDAP, они завершаются с ошибкой, поскольку создание пользователей отключено.
Аналогично завершается с ошибкой создание пользователей на основе SAML и других провайдеров OmniAuth.

{% alert level="info" %}
Когда создание пользователя блокируется при входе через LDAP, пользователь видит сообщение о том, что инстанс находится
в режиме обслуживания (только для чтения), а не общую ошибку «доступ запрещён».
{% endalert %}

### Действия с Git

Все операции чтения Git продолжают работать, например `git clone` и `git pull`.
Все операции записи завершаются с ошибкой — как через CLI, так и через Web IDE — с сообщением:
`Git push is not allowed because this instance is currently in (read-only) maintenance mode.`

### Запросы на слияние, задачи

Все операции записи, кроме упомянутых выше, завершаются с ошибкой.
Например, пользователь не может обновить запрос на слияние или задачу.

### Входящая почта

Создание ответов на задачи, новых задач (включая задачи Service Desk) и запросов на слияние по электронной почте завершается с ошибкой.
Обработчики входящей почты пропускают обработку, пока включён режим обслуживания.

### Исходящая почта

Уведомления по электронной почте продолжают приходить, но письма, требующие записи в базу данных
(например, сброс пароля), не доставляются.

### REST API

Для большинства JSON-запросов методы `POST`, `PUT`, `PATCH` и `DELETE` блокируются, и API возвращает ответ `503`
с сообщением об ошибке `system is in maintenance mode` и заголовком `Retry-After`.

Разрешены только следующие запросы:

| HTTP-запрос | Разрешённые маршруты | Примечания |
| ----------- | -------------------- | ---------- |
| `POST` | `/admin/application_settings/general` | Чтобы разрешить обновление настроек приложения в интерфейсе администратора. |
| `PUT` | `/api/v4/application/settings` | Чтобы разрешить обновление настроек приложения через API. |
| `POST` | `/users/sign_in` | Чтобы разрешить вход пользователей. |
| `POST` | `/users/sign_out` | Чтобы разрешить выход пользователей. |
| `POST` | `/oauth/token` | Чтобы разрешить получение OAuth-токенов. |
| `POST` | `/admin/session`, `/admin/session/destroy` | Чтобы разрешить режим администратора (Admin Mode). |
| `POST` | Пути, оканчивающиеся на `/compare` | Маршруты сравнения ревизий Git. |
| `POST` | `*.git/git-upload-pack` | Чтобы разрешить Git pull/clone. |
| `POST` | `/api/v4/internal` | Маршруты внутреннего API. |
| `POST` | `/admin/sidekiq` | Чтобы разрешить управление фоновыми задачами в разделе **Admin**. |

### GraphQL API

Запросы `POST /api/graphql` разрешены, но мутации блокируются с сообщением об ошибке
`You cannot perform write operations on a read-only instance`.

### Непрерывная интеграция

- Новые задачи и конвейеры не запускаются — ни по расписанию, ни вручную.
- Задачи, которые уже выполнялись, продолжают отображаться со статусом `running` в интерфейсе, даже если они завершились на раннере.
- Задачи в состоянии `running` дольше лимита времени проекта не завершаются по таймауту.
- Конвейеры нельзя запустить, перезапустить или отменить. Новые задачи также не создаются.
- Статус раннеров в **Admin** → **Раннеры** не обновляется.

После отключения режима обслуживания новые задачи снова начинают подбираться.
Задачи, которые были в состоянии `running` до включения режима обслуживания, возобновляются, и их логи снова обновляются.

{% alert level="info" %}
Перезапустите ранее выполнявшиеся (`running`) конвейеры после отключения режима обслуживания.
{% endalert %}

### Развёртывания

Развёртывания не выполняются, поскольку конвейеры не завершены.
Отключите автоматические развёртывания на время режима обслуживания и включите их после его отключения.

### Реестр контейнеров

`docker push` завершается с ошибкой `denied: requested access to the resource is denied`, но `docker pull` работает.

### Реестр пакетов

Реестр пакетов позволяет устанавливать пакеты, но не публиковать их.

### Фоновые задачи

Фоновые задачи (cron-задачи, Sidekiq) продолжают выполняться как обычно, поскольку они не отключаются автоматически.
Так как фоновые задачи выполняют операции, которые могут изменить внутреннее состояние инстанса, на время режима
обслуживания может потребоваться отключить часть из них или все.

Чтобы отслеживать очереди и отключать задачи:

1. Перейдите в **Admin** → **Мониторинг** → **Фоновые задачи**.
1. В панели Sidekiq выберите **Cron** и отключите задачи по отдельности или все сразу, нажав **Disable All**.

### Управление инцидентами

Функции управления инцидентами ограничены.
Создание алертов и инцидентов полностью приостановлено.
Соответственно, уведомления и оповещения по алертам и инцидентам отключены.

### Функциональные флаги

- Функциональные флаги разработки нельзя включать или отключать через API, но можно переключать через консоль Rails.
- Сервис функциональных флагов отвечает на проверки флагов, но переключать сами флаги нельзя.
