CodeScoring — это инструмент анализа состава программного обеспечения (SCA/OSA) для проверки сторонних зависимостей на наличие уязвимостей, лицензионных рисков и нарушений политик безопасности.

Интеграция CodeScoring в Deckhouse Code позволяет использовать возможности CodeScoring в CI/CD-пайплайнах и покрывает следующие SCA/OSA-сценарии:

- анализ зависимостей;
- обнаружение уязвимостей;
- генерация SBOM;
- разбор результатов на стороне платформы.

Статический (SAST) и динамический анализ (DAST) в эту интеграцию не входят.

## Возможности интеграции

Интеграция предоставляет следующие возможности:

- анализ зависимостей приложения (пакеты, библиотеки, версии);
- обнаружение уязвимостей по базам CVE (включая ФСТЭК БДУ и Kaspersky Open Source Software Threats Data Feed);
- генерация SBOM в формате CycloneDX;
- формирование нативных отчётов GitLab за один прогон: «Dependency Scanning», «Code Quality», «JUnit», а также «SARIF» и «CycloneDX SBOM»;
- анализ результатов и применение политик (пороги severity, игнорирование срабатываний) на стороне платформы CodeScoring.

## Предварительные требования

Перед настройкой интеграции убедитесь, что:

- развёрнут сервер CodeScoring (on-premise или SaaS);
- получен API-токен в профиле пользователя CodeScoring;
- в проекте или группе доступен GitLab Runner с executor `docker`. Задача сканера выполняется в контейнере `debian:bookworm-slim`.

{% alert level="info" %}
Консольный агент Johnny не требуется устанавливать вручную. Он скачивается автоматически с вашего сервера CodeScoring по API-токену при каждом запуске задачи сканера.
{% endalert %}

## Развёртывание сервера CodeScoring

Требования и варианты установки сервера CodeScoring описаны в официальной документации:

- [системные требования](https://docs.codescoring.ru/admin-guide/server-requirements);
- [установка в Docker](https://docs.codescoring.ru/admin-guide/installation);

## Настройка интеграции в проекте

Параметры подключения к CodeScoring задаются через настройки проекта или группы:

1. Откройте проект в Deckhouse Code.
1. Перейдите в раздел «Настройки» → «Интеграции» → «CodeScoring».
1. Заполните параметры подключения:

   | Параметр | Описание |
   |----------|----------|
   | «Активна» | Переключатель включения интеграции для проекта |
   | «URL сервера» | Адрес сервера CodeScoring. Например, `https://codescoring.example.com` |
   | «API-токен» | Токен из профиля пользователя CodeScoring (хранится в зашифрованном виде и маскируется) |
   | «CA-сертификат» | Необязательный PEM-сертификат CA для сервера CodeScoring с самоподписанным сертификатом |
   | «Название проекта» | Имя проекта в CodeScoring (по умолчанию — строковый идентификатор (slug) репозитория) |
   | «Стадия сканирования» | Стадия для привязки результатов на стороне платформы (по умолчанию `build`) |

1. Нажмите «Сохранить».

Интеграция автоматически передаёт в пайплайн следующие CI-переменные (задавать их вручную в `.gitlab-ci.yml` не требуется):

- `FE_SCANS_CODESCORING_URL`;
- `FE_SCANS_CODESCORING_TOKEN`;
- `FE_SCANS_CODESCORING_CA_CERT`;
- `FE_SCANS_CODESCORING_PROJECT`;
- `FE_SCANS_CODESCORING_SCAN_STAGE`.

## Запуск сканирования

Сканер подмешивается в пайплайн через **scan-execution-политику**, а не ручным `include`.

Для настройки автоматического сканирования выполните следующее:

1. В проекте политик безопасности добавьте в `policy.yml` действие `codescoring`:

   ```yaml
   scan_execution_policy:
     - name: CodeScoring on every pipeline
       enabled: true
       rules:
         - type: pipeline
           branches: ["*"]
       actions:
         - scan: codescoring
   ```

1. Привяжите проект политик к целевому проекту в разделе «Настройки» → «Security policy».

После этого в каждый пайплайн автоматически добавляется задача `codescoring_scan` (стадия `fe-security-scanner`), которая:

- скачивает агент Johnny с сервера CodeScoring (по токену; при самоподписанном сертификате — с переданным CA-сертификатом);
- сканирует рабочую директорию и отдаёт нативные отчёты GitLab.

Отдельный `include` и ручная установка переменных `CODESCORING_*` не требуются. Всё необходимое подставляют интеграция и политика.

## Отчёты и просмотр результатов

Одна задача `codescoring_scan` формирует все отчёты за один прогон. Deckhouse Code основан на GitLab FOSS, где часть EE-виджетов отсутствует, поэтому результаты выводятся в следующем виде:

| Отчёт | Где смотреть |
|-------|--------------|
| Тесты (JUnit) | Вкладка «Tests» пайплайна (нативно) |
| Code Quality | Виджет «Code Quality» в запросе на слияние (нативно) |
| Dependency Scanning | Страница «CodeScoring» по адресу `/-/security/codescoring` |
| SBOM (состав зависимостей) | Страница «Dependency list» по адресу `/-/security/dependencies` |
| Лицензии | Страница «License compliance» по адресу `/-/security/licenses` |
| SARIF | Загружается артефактом (отдельного SAST-виджета в FOSS нет) |

{% alert level="info" %}
Страницы «Dependency Scanning», «Dependency list» и «License compliance» — это FE-реализация Deckhouse Code. В GitLab FOSS соответствующие виджеты доступны только в EE. Сейчас страницы открываются по прямому URL-адресу (в будущих версиях Code появится пункт в боковом меню).
{% endalert %}

## Политики и блокировка

Задача `codescoring_scan` не является блокирующей. Она всегда завершается успешно и загружает отчёты (в том числе на упавших попытках — `artifacts:when: always`), не приводя к сбою пайплайна.

Настройка политик (40 критериев, пороги критичности (severity), разбор) и решение о блокировке выполняются на стороне платформы CodeScoring. Жёсткая блокировка пайплайна по нарушению политики — предмет отдельной настройки scan-execution-политики и в текущем шаблоне не включена.

## Разбор уязвимостей

Обнаруженные уязвимости можно разбирать непосредственно в интерфейсе CodeScoring. Для этого:

1. Перейдите в раздел «SCA» → «Уязвимости».
1. Выберите статус: `Активен`, `Подтверждён`, `Не затронут` или `Ложноположительный`.
1. Заполните обоснование и ответ (совместимо с форматом CycloneDX VEX).

Можно настроить временное игнорирование срабатываний — по проекту, технологии, пакету, лицензии или CVE.

{% alert level="warning" %}
На текущем этапе агент CodeScoring не заполняет поле `severity` в «Dependency Scanning Report» (только в «Code Quality»), поэтому на странице CodeScoring уровень критичности может отображаться как `unknown`.
{% endalert %}

## Устранение неполадок

### Сканирование не запускается

Проверьте следующее:

- интеграция CodeScoring активна в настройках проекта (заданы URL-адрес и токен);
- проект политик привязан к проекту и в нём задано действие `- scan: codescoring`;
- в пайплайне присутствует задача `codescoring_scan` и доступен GitLab Runner с executor `docker`.

### Результаты не появляются на страницах CodeScoring, Dependency list, License compliance

Проверьте следующее:

- задача `codescoring_scan` завершилась и загрузила артефакты `gl-dependency-scanning-report.json` и `gl-sbom.cdx.json` (артефакты собираются при `when: always`);
- открывается страница ветки по умолчанию (страницы читают отчёт последнего пайплайна).

### Виджет Code Quality не отображается в запросе на слияние

Проверьте, что после выполнения задачи был сформирован `gl-code-quality-report.json` и объявлен в секции `artifacts:reports:codequality`.
