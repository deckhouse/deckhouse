Внешние проверки статуса показывают результат проверок, которые выполняются сервисами вне Deckhouse Code. Maintainer проекта может настроить эти проверки для проекта и определить целевые ветки, к которым они применяются.

Используйте внешние проверки статуса на странице запроса на слияние, чтобы понять, подтвердили ли внешние системы текущее состояние запроса.

## Как работают внешние проверки статуса

Когда происходит событие запроса на слияние, Deckhouse Code отправляет информацию о запросе во все внешние проверки статуса, которые применяются к целевой ветке. Внешний сервис проверяет запрос на слияние и отправляет результат обратно в Deckhouse Code.

Результат проверки всегда относится к текущему `HEAD` SHA source branch. Если в запрос на слияние добавлены новые коммиты, предыдущий результат больше не относится к новому состоянию, и Deckhouse Code ожидает новый результат.

Внешние проверки статуса влияют на возможность слияния только если в проекте включена настройка **Status checks must succeed**. Если настройка выключена, виджет продолжает показывать результаты проверок, но failed- или pending-проверки не блокируют слияние.

## Виджет External status checks

Виджет **External status checks** отображается на странице запроса на слияние, если для запроса есть применимые внешние проверки.

Виджет показывает:

- Название проверки.
- Статус проверки.
- URL внешнего сервиса, если ваша роль позволяет его видеть.
- Подробности ошибки для failed-проверок, если Deckhouse Code сохранил причину ошибки.
- Действие **Retry** для failed-проверок, если ваша роль позволяет их перезапускать.

Виджет помогает авторам и ревьюерам понять, какие внешние системы еще должны ответить до слияния запроса.

## Статусы проверок

Внешние проверки статуса в виджете могут иметь следующие статусы:

| Статус | Описание |
|--------|----------|
| `pending` | Deckhouse Code ожидает ответ внешнего сервиса. |
| `passed` | Внешний сервис подтвердил текущее состояние запроса на слияние. |
| `failed` | Внешний сервис отклонил запрос на слияние, запрос к внешнему сервису завершился ошибкой или истек таймаут. |

Если проверка находится в состоянии `failed`, наведите указатель на failed-проверку, чтобы посмотреть причину ошибки. Причина может указывать на ошибку внешнего сервиса, проблему подключения, заблокированный URL или автоматическое завершение по таймауту.

## Pending-проверки

Проверка остается в состоянии `pending`, пока Deckhouse Code ожидает ответ внешнего сервиса.

Пока хотя бы одна проверка находится в состоянии `pending`, виджет запроса на слияние обновляет внешние проверки статуса примерно каждые 10 секунд. Polling останавливается, когда проверок в состоянии `pending` больше нет.

Pending-проверка может автоматически перейти в состояние `failed`, если внешний сервис не ответит до истечения настроенного таймаута. Таймаут настраивается на уровне проекта или для конкретной внешней проверки статуса.

## Failed-проверки

Проверка может перейти в состояние `failed`, если:

- Внешний сервис вернул `failed` для текущего `HEAD` SHA запроса на слияние.
- Deckhouse Code не смог отправить запрос во внешний сервис.
- URL внешнего сервиса заблокирован или недоступен.
- Внешний сервис не ответил до истечения таймаута.

Если в проекте включена настройка **Status checks must succeed**, failed-проверка блокирует слияние до тех пор, пока проверка не перейдет в `passed` или пока настройки проекта не будут изменены.

## Перезапуск failed-проверки

Вы можете перезапустить failed-проверку, если у вас есть роль **Developer** или выше в проекте.

Перезапуск доступен только для failed-проверок, которые относятся к текущему `HEAD` SHA source branch запроса на слияние.

Чтобы перезапустить failed-проверку:

1. Откройте запрос на слияние.
1. Найдите виджет **External status checks**.
1. В строке failed-проверки нажмите **Retry**.

После перезапуска Deckhouse Code переводит проверку обратно в состояние `pending` и повторно отправляет текущий payload запроса на слияние во внешний сервис.

Используйте перезапуск после исправления проблемы во внешнем сервисе или если сбой был временным.

## URL проверок

Виджет может показывать URL внешнего сервиса для проверки. URL виден только пользователям, роль которых позволяет смотреть URL внешних проверок статуса. В стандартных ролях проекта это доступно пользователям с ролью **Developer** или выше.

Если вы не видите URL, вы все равно можете видеть статус проверки, если ваша роль позволяет смотреть ответы проверок статуса.

## Доступ к результатам проверок

Доступ к информации о внешних проверках статуса зависит от роли в проекте:

| Действие | Минимальная роль |
|----------|------------------|
| Смотреть ответы проверок в виджете | **Reporter** |
| Смотреть URL внешнего сервиса | **Developer** |
| Перезапустить failed-проверку | **Developer** |

Для internal projects аутентифицированные пользователи, которые могут читать запросы на слияние в целевом проекте, также могут смотреть ответы проверок статуса.

## Область применения по веткам

В проекте могут быть проверки, которые применяются к:

- Всем веткам.
- Всем защищенным веткам.
- Выбранным защищенным веткам.

Если целевая ветка запроса на слияние не соответствует области применения проверки, проверка не отображается в виджете запроса на слияние.

Когда целевая ветка меняется, Deckhouse Code пересчитывает применимые проверки. Проверки, которые больше не применяются, удаляются из запроса на слияние, а новые применимые проверки добавляются в состоянии `pending`.

## Troubleshooting

### Запрос на слияние заблокирован внешней проверкой статуса

Проверьте:

- Включен ли флажок **Status checks must succeed**.
- Применяется ли проверка к целевой ветке запроса на слияние.
- Отправил ли внешний сервис `passed` для текущего `HEAD` SHA.
- Не истек ли таймаут до callback внешнего сервиса.

### Проверка завершилась с failed по таймауту

Проверьте:

- Значение проекта **Status checks timeout**.
- Индивидуальное значение **Timeout minutes** у проверки.
- Время ответа внешнего сервиса.
- Нужно ли перезапустить проверку после исправления внешнего сервиса.

### Проверка остается в pending

Проверка может оставаться в состоянии `pending`, пока Deckhouse Code ожидает внешний сервис.

Проверьте:

- Доступен ли внешний сервис.
- Может ли сервис обработать payload запроса на слияние.
- Отправляет ли сервис ответ для текущего `HEAD` SHA.
- Достаточен ли настроенный таймаут для завершения обработки.

Если таймаут истечет, проверка перейдет в состояние `failed`.

### Запрос во внешний сервис завершился ошибкой

Проверьте:

- URL в поле **API to check**.
- Сетевой доступ от Deckhouse Code до внешнего сервиса.
- Использует ли URL протокол `http` или `https`.
- Текст ошибки в виджете запроса на слияние.

### Retry недоступен

Действие **Retry** отображается только если выполняются все условия:

- Проверка находится в состоянии `failed`.
- Проверка относится к текущему `HEAD` SHA запроса на слияние.
- У вас есть роль **Developer** или выше в проекте.

Если retry недоступен, попросите maintainer проекта проверить настройки проекта и вашу роль.

### URL внешнего сервиса не отображается

URL скрыт, если ваша роль не позволяет смотреть URL внешних проверок статуса. Если вам нужен доступ к URL внешнего сервиса, обратитесь к maintainer проекта.
